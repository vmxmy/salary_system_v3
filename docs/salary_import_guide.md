
# 工资数据导入约束与验证指南

## 1. 目的

本文档旨在为工资数据的导入流程提供一套严格的约束和标准的验证程序。遵循本指南是确保薪酬系统数据准确性、一致性和可追溯性的关键。所有数据导入操作都必须严格遵守本文档定义的规则。

## 2. 核心原则

1.  **只导入明细，不导入合计**: 系统的核心设计思想是，所有合计金额（如应发、扣发、实发）都必须由明细数据计算得出。原始数据源中的任何合计字段都应被忽略。
2.  **收入项驱动，而非原始列名驱动**: 数据的唯一事实来源是 `salary_components` 表中定义的工资项。导入的源数据（如CSV）中的列名必须与 `salary_components` 中的名称建立明确的映射关系。
3.  **批次导入，批次验证**:每一次独立的数据导入任务（例如，导入某个月的工资）都必须被视为一个独立的批次。每个批次在完成后都必须经过严格的验证流程，确认无误后方可视为最终完成。

## 3. 详细导入流程

### 步骤一：数据映射与准备

1.  **识别数据源**: 确定要导入的源文件（例如 `2025年1月应发及五险一金扣除费用.csv`）。
2.  **识别收入项**: 从源文件中，仅识别出所有属于**工资收入类**的明细字段。所有扣款项、合计项、统计项都必须被忽略。
3.  **建立映射关系**:
    *   将识别出的每个收入项列名与 `public.salary_components` 表中的 `name` 字段进行精确匹配。
    *   允许存在合理的名称差异（如CSV中包含空格 `级别/岗位级别 工资` vs. 数据库中为 `级别/岗位级别工资`），但必须在导入逻辑中建立一个明确的映射表。
    *   **严禁**导入任何在 `salary_components` 表中找不到对应项的列。如果源文件出现了新的工资项，必须先在 `salary_components` 表中定义该项，然后再进行导入。

**映射表示例:**

| CSV 源数据列名 | 数据库 `salary_components.name` |
| :--- | :--- |
| `职务/技术等级 工资` | `职务/技术等级工资` |
| `级别/岗位级别 工资` | `级别/岗位级别工资` |
| `93年工改保留补贴` | `九三年工改保留津补贴` |
| `公务员规范性津贴补贴` | `公务员规范后津补贴` |
| `见习试用期工资` | `试用期工资` |
| `月基础绩效` | `基础绩效` |
| `信访工作人员岗位津贴` | `信访工作人员岗位工作津贴` |
| ... | ... |

### 步骤二：执行导入

1.  **暂存数据**: 将源CSV文件中经过映射的收入类明细数据，连同用于匹配员工的唯一标识（**身份证号**），插入到一个临时的暂存表（如 `temp_january_salary_import`）中。
2.  **创建工资单主记录**:
    *   基于暂存表中的员工身份证号，关联 `public.employees` 表，找到对应的 `employee_id`。
    *   为该批次的每个员工，在 `public.payrolls` 表中创建一条主记录。此时，`gross_pay`, `total_deductions`, `net_pay` 等合计字段应为0或NULL。
3.  **插入工资明细**:
    *   将暂存表中的宽表数据转换为长表结构。
    *   对于每一个工资项，都在 `public.payroll_items` 表中插入一条记录，包含对应的 `payroll_id`、`component_id` 和 `amount`。

### 步骤三：导入后验证

这是整个流程中最关键的一步，必须在每次导入批次后立即执行。

1.  **计算应发合计**: 使用以下逻辑的SQL查询，从刚刚导入的 `payroll_items` 表中，按员工计算出应发工资的总和。

    ```sql
    -- 概念验证查询
    SELECT
        e.id_number, -- 员工身份证号，用于与CSV文件核对
        SUM(pi.amount) AS calculated_gross_pay -- 计算出的应发合计
    FROM
        public.payrolls p
    JOIN
        public.payroll_items pi ON p.id = pi.payroll_id
    JOIN
        public.salary_components sc ON pi.component_id = sc.id
    JOIN
        public.employees e ON p.employee_id = e.id
    WHERE
        p.pay_period_start = '2025-01-01' -- **必须用导入批次的周期进行筛选**
        AND sc.type = 'earning' -- **只计算收入类型的项**
    GROUP BY
        e.id_number;
    ```

2.  **比对源数据**:
    *   将上述查询结果中的 `calculated_gross_pay` 与源CSV文件中对应员工的 `应发工资` 列进行逐一比对。
    *   可以使用脚本自动化此比对过程，或在小批量导入时进行人工抽查。
    *   **验证标准：两者必须完全相等，精确到小数点后两位。**

3.  **处理差异**:
    *   若存在任何差异，必须暂停后续流程（如更新合计字段、删除临时表等）。
    *   必须彻查差异原因，可能是映射错误、数据类型问题或计算逻辑遗漏。
    *   在问题解决并重新验证通过之前，该批次导入被视为**失败**。

4.  **完成导入**:
    *   只有在验证完全一致后，才能继续执行后续步骤，例如更新 `payrolls` 表中的合计字段，并清理临时表。

## 4. 强制性约束总结

-   **严禁**: 直接向 `payrolls` 表的合计字段 (`gross_pay`, `net_pay` 等) 插入来自源文件的值。
-   **严禁**: 导入任何扣发类项目到 `payroll_items` 表。扣发逻辑应由系统根据社保、税务等规则独立计算。
-   **必须**: 确保每一个被导入的CSV列都在 `salary_components` 中有明确的、经过验证的映射。
-   **必须**: 在每次导入批次后，严格执行与源文件的“应发合计”验证流程。
